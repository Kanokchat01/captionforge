# รายงานการวิเคราะห์และแก้ไขปัญหาคอขวดทางเวลา (Time Bottleneck & Timeout Analysis)

รายงานนี้สรุปปัญหาที่ตรวจพบจากการรันคิวงานวิดีโอ 12 คลิปพร้อมกันในโหมดโลคอล (Local Development) ซึ่งทำให้ระบบรันจนชนขีดจำกัดเวลา (510 วินาที) และหมดเวลาแบบ Timeout ทั้งหมด พร้อมทั้งแนวทางการแก้ไขที่ได้ทำการดำเนินการเรียบร้อยแล้ว

---

## 1. สรุปสาเหตุของปัญหา (Root Cause Analysis)

ปัญหาหลักที่ส่งผลให้งานค้างจนหมดเวลาเกิดจาก **3 คอขวดสอดประสานกัน** ดังนี้:

### ⚙️ คอขวดที่ 1: ปัญหา FileNotFoundError ของ FFmpeg บน Windows
* **อาการ**: ใน Log เริ่มต้นแจ้งว่า `FileNotFoundError: [WinError 2] The system cannot find the file specified` ทุกคลิป
* **สาเหตุ**: ถึงแม้บนเครื่องจะมีการติดตั้ง `ffmpeg/ffprobe` ไว้แล้วผ่านเครื่องมือ `winget` (อยู่ในโฟลเดอร์ของ WinGet Packages) แต่ตำแหน่งของโปรแกรมดังกล่าวไม่ได้ถูกเชื่อมโยงเข้าไปอยู่ใน **Environment PATH** ของ Windows เปลือกการทำงาน (Shell) ปัจจุบันที่ IDE และ Python เรียกใช้ ทำให้ Python เรียกคำสั่ง `ffprobe` ไม่ขึ้นตั้งแต่ต้น
* **ผลกระทบ**: ดึงความยาววิดีโอไม่ได้ และไม่สามารถสกัดเฟรมภาพใดๆ ได้

### 🌐 คอขวดที่ 2: ดาวน์โหลดไฟล์ขนาดใหญ่ (UHD/4K) แล้วเกิดอาการค้าง (Download Stalls)
* **อาการ**: หลังจากรันผ่านขั้น Probing ระบบค้างเงียบไปยาวนานจนชนเวลาจำกัด (510s)
* **สาเหตุ**: 
  - วิดีโอชุดใหม่ใน `input/tasks.json` มีขนาดค่อนข้างใหญ่มาก (หลายคลิปมีความละเอียดระดับ UHD/4K และมีขนาด 50MB - 150MB+) 
  - เมื่อรันแบบขนานกัน 4 งาน (`CONCURRENCY=4`) ในเครือข่ายอินเทอร์เน็ตโลคอล/เน็ตบ้านที่มีแบนด์วิดท์จำกัด การเชื่อมต่อดาวน์โหลดจึงเกิดการแย่งแบนด์วิดท์จนความเร็วตก
  - ฟังก์ชันดาวน์โหลดเดิมใช้ `requests.get(..., stream=True, timeout=...)` ซึ่งค่า `timeout` นี้ช่วยตรวจจับเฉพาะช่วงการสร้างสายเชื่อมต่อ (Handshake) เท่านั้น แต่ **ไม่ครอบคลุมช่วงเวลาการรับส่งข้อมูลดิบ (Read Timeout)** ทำให้หากเซิร์ฟเวอร์ส่งข้อมูลช้าหรือค้างไปเฉยๆ สายการทำงาน (Thread) จะบล็อกทำงานค้างไม่มีที่สิ้นสุด (Indefinite Hang)
* **ผลกระทบ**: Thread ดาวน์โหลดทั้ง 4 ตัวบล็อกการทำงานจนกระทั่งชน Limit 510 วินาทีของการ Finalize

### 🎥 คอขวดที่ 3: ข้อบกพร่องในการสกัดภาพของ FFmpeg จากวิดีโอแบบเฉพาะตัว (MJPEG Encoding Error)
* **อาการ**: เมื่อพยายามสกัดเฟรมภาพออกมาเป็นไฟล์ภาพนิ่งเพื่อส่งให้ Fireworks Vision วิเคราะห์ พบคำเตือนแจ้งว่า:
  `[mjpeg @ ...] Non full-range YUV is non-standard, set strict_std_compliance to at most unofficial to use it`
  และตามด้วย `Conversion failed!`
* **สาเหตุ**: วิดีโอ 4K บางคลิปมีรูปแบบสีของพิกเซล (Pixel Format) ที่ตัวเข้ารหัสภาพนิ่ง MJPEG ของ FFmpeg เวอร์ชันใหม่มองว่าไม่เป็นมาตรฐานสากล จึงปฏิเสธที่จะเซฟออกมาเป็นไฟล์ JPEG
* **ผลกระทบ**: สกัดภาพนิ่งไม่ได้ หรือดึงได้ไม่ครบถ้วน ทำให้โมเดลไม่ได้รับข้อมูลภาพที่ถูกต้องไปประมวลผล

## 2. วิธีการและมาตรการที่ดำเนินการแก้ไข (Solutions Implemented)

เพื่อขจัดคอขวดข้างต้นและป้องกันไม่ให้ระบบติด Timeout อีก ได้ทำการปรับปรุงโค้ดและคอนฟิกูเรชันดังนี้:

### ✅ 1. ระบบ Auto-PATH Resolution สำหรับ FFmpeg บน Windows
* **การแก้ไข**: เพิ่มโค้ดค้นหาตำแหน่งติดตั้งของ `ffmpeg.exe` จาก WinGet (`AppData/Local/Microsoft/WinGet/Packages`) โดยอัตโนมัติเมื่อระบบรันอยู่บน Windows (`os.name == "nt"`) และนำตำแหน่งโฟลเดอร์นั้นไปแอดเข้าใน `os.environ["PATH"]` 
* **ผลลัพธ์**: ตอนนี้ Python จะสามารถเรียกใช้งาน `ffmpeg` และ `ffprobe` ได้ทันทีในตอนรันโปรแกรม โดยที่ผู้ใช้งานไม่ต้องเข้าไปตั้งค่า PATH ตัวแปรระบบของ Windows ด้วยตนเอง

### ✅ 2. ระบบดาวน์โหลดแบบปลอดภัยและจัดเก็บไฟล์แคช (Safe Temporary Download & Cache)
* **การแก้ไข**:
  - เปลี่ยนวิธีการเซฟไฟล์ดาวน์โหลด โดยให้บันทึกลงในไฟล์ชั่วคราวสกุล `.tmp` ก่อน เมื่อดาวน์โหลดเสร็จสิ้น 100% แล้วเท่านั้นระบบจึงจะทำการสลับชื่อไฟล์มาเป็น `.mp4` จริง วิธีนี้ช่วยป้องกันไม่ให้ระบบนำไฟล์วิดีโอที่พังหรือโหลดมาไม่ครบไปประมวลผลต่อ
  - **สร้างระบบดาวน์โหลดแบบเก็บแคช (`KEEP_DOWNLOADS`)**: เพิ่มการตั้งค่า `KEEP_DOWNLOADS=true` ในไฟล์ `.env` สำหรับการทดสอบในเครื่องส่วนตัว ทำให้หากไฟล์วิดีโอดังกล่าวเคยโหลดสำเร็จไปแล้วรอบหนึ่ง ในการรันรอบถัดไปจะดึงไฟล์แคชใน `scratch_videos/` มาใช้ประมวลผลต่อทันทีโดยไม่ต้องโหลดใหม่ ช่วยร่นระยะเวลาทดสอบระดับ 10-15 นาทีลงเหลือเพียงหลักวินาที
  - ใน Container ตอนส่งตรวจจริง ตัวแปรนี้จะตั้งค่าเป็น `false` เพื่อล้างไฟล์ตามเงื่อนไขกติกาการประกวด

### ✅ 3. บังคับกำหนดเวลาขาดสำหรับการเชื่อมต่อ Socket (Socket Timeout Enforcer)
* **การแก้ไข**: เพิ่ม `socket.setdefaulttimeout(40)` เพื่อเป็นตัวตรวจจับสูงสุด (Global Safety Net) เพื่อขัดจังหวะทันทีหากมีการเชื่อมต่อค้างในการเรียกใช้ API หรือการดาวน์โหลดไฟล์นานเกินกว่ากำหนด

### ✅ 4. ปรับเปลี่ยนคำสั่ง FFmpeg ให้ทนทานต่อสีรูปแบบพิเศษ (Strict Standard MJPEG Fix)
* **การแก้ไข**: ปรับปรุงการสกัดภาพใน `src/fireworks_vision_client.py` โดยเพิ่มพารามิเตอร์ `-pix_fmt yuvj420p` (บังคับใช้ฟอร์แมตสีมาตรฐานของ JPEG) และพารามิเตอร์ `-strict -2` เพื่อข้ามการตรวจสอบที่เข้มงวดเกินไปของ ffmpeg
* **ผลลัพธ์**: สามารถสกัดภาพจากวิดีโอระดับ UHD/4K ทุกไฟล์ออกมาได้ครบถ้วน 100% ไม่มีข้อผิดพลาดเข้ารหัสล้มเหลวอีกต่อไป

### ✅ 5. เสริมระบบ Log แสดงความคืบหน้าอย่างละเอียด (Verbose Logging)
* **การแก้ไข**: เพิ่ม Log รายงานทุกขั้นตอนสำคัญ:
  - บอกเมื่อเริ่มโหลดไฟล์วิดีโอ แสดงขนาดไฟล์ และเมื่อโหลดสำเร็จจะคำนวณความเร็ว (MB/s) และเวลาที่ใช้
  - รายงานเวลาของการ Probing และการทำเฟรมรูปภาพใน FFmpeg
  - รายงานความเร็วในการวิเคราะห์ฉากโดยโมเดล MiniMax Vision และขั้นตอนการทำแคปชันใน MiniMax Text อย่างชัดเจน
