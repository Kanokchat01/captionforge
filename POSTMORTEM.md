AMD Developer Hackathon Act II · Track 2 · ทีม M3OWAI · 11–13 ก.ค. 2026

เอกสารนี้บันทึกว่าคะแนนร่วงจาก 0.90 ไป 0.66 เพราะอะไร แก้อะไรถึงกลับขึ้นไป 0.91
และบทเรียนที่ควรเอาไปใช้ครั้งหน้า

## 2. วินิจฉัย: เลขมันบอกเองว่าเกิดอะไรขึ้น

`main.py` มี generic fallback caption กันคลิปพัง — มัน**ได้คะแนน style ~0.5 แต่ accuracy ~0**
เพราะไม่ได้พูดถึงวิดีโอจริงเลย ⇒ คลิปที่ตก fallback ได้ประมาณ **0.25** แทนที่จะเป็น **0.91**

แทนค่าในสูตร `0.91 − (k/12) × 0.66 = คะแนนที่ได้`:

| คะแนนบอร์ด | k = จำนวนคลิปที่ตาย |
|---|---|
| 0.69 (r6) | ≈ **4.0** |
| 0.66 (r1) | ≈ **4.5** |

**สองรอบติดกันตกลงมาเกาะเลขที่ตรงกับ "เสียคลิปไป 4–5 จาก 12"**

นี่คือจุดตัดสินใจที่สำคัญที่สุดของทั้งงาน:

> ถ้า **judge โหดขึ้น** จริง คะแนนจะลดแบบ**เรียบ ๆ ทุก caption** (เห็น 0.78, 0.80)
> มันไม่บังเอิญไปตรงกับเลขคณิตของ "คลิปตายทั้งคลิป" ถึงสองครั้ง
>
> ⇒ **นี่คือคลิปตาย ไม่ใช่คำบรรยายห่วย**

ถ้าตอนนั้นเชื่อว่า judge เปลี่ยน เราคงไปนั่งรื้อ prompt 4 ชั่วโมง แล้วก็จะยังติดอยู่ที่ 0.66

## 3. ทำไมคลิปถึงตาย: เรายิง API ถี่กว่าทีมที่ชนะ 4–7 เท่า

อ่านซอร์สจริงของ **SwiftCap** (0.91) เทียบกับเรา:

| | SwiftCap (0.91) | CaptionForge r1 (0.66) |
|---|---|---|
| โมเดล | `minimax-m3` | `qwen3p7-plus` |
| call ต่อคลิป | **1** (+retry 1) | **4** (แยกสไตล์) |
| call รวม 12 คลิป | **12–24** | **48–90** (รวม retry/spare/regen) |
| call ค้างในอากาศสูงสุด | **1** — loop แบบ sequential | **16** (CONCURRENCY 4 × 4 สไตล์) |
| เฟรม | stream 1fps ≤512px ไม่ download | download, 4 เฟรม @1024px |

และวันนั้นเรา **เห็น 429 ของจริงจาก Fireworks เต็มไปหมด** ตอนรัน local

บนเครื่องตรวจ (เน็ตเร็ว ไม่มี download ช้า ๆ มาช่วยถ่างจังหวะ) call ทั้ง 16 ยิงพร้อมกัน →
เผา retry `[3, 6]` จนหมด → spare model ก็โดน 429 ต่อ → r1 **ไม่มีขั้นบันไดกู้ต่ำกว่านั้น** →
สไตล์นั้นตก generic fallback → คิดเป็น 4–5 คลิป → **0.66**

## 4. สิ่งที่แก้ (v8)

### 4.1 ตัวที่ทำให้คะแนนขึ้นจริง — ลดจำนวน call

เปลี่ยน engine จาก `qwen_direct` → **`minimax_single`**: ยิง **1 call ต่อคลิป** ได้ครบ 4 สไตล์
เป็น JSON ก้อนเดียว + ลด `CONCURRENCY` จาก 4 → **2**

| | r1 | v8 |
|---|---|---|
| call ต่อคลิป | 4 | **1** |
| call รวมต่อรอบ | 48–90 | **12–15** |
| call ค้างในอากาศ | 16 | **2** |
| โมเดล | qwen3p7-plus | qwen3p7-plus (**ตัวเดิม**) |
| prompt / persona | r1 | **r1 เป๊ะ — ไม่แตะเลย** |

**เปลี่ยนแค่ "ห่อของยังไง" ไม่ได้เปลี่ยน "เขียนอะไร"**

ผลข้างเคียงที่ดี: ทุกสไตล์เขียนจากหลักฐานชุดเดียวกันในลมหายใจเดียว ⇒ 4 สไตล์ขัดแย้งกันเองไม่ได้

**ทำไมยังใช้ qwen3p7-plus ไม่ใช่ minimax-m3 ตาม SwiftCap:** ทดสอบ head-to-head 15 คลิป —
judge แยกไม่ออก (0.983 vs 0.988) แต่ **minimax ต้อง retry แก้ style 3 ครั้ง ส่วน qwen ครั้งเดียว
และช้ากว่า 2.3 เท่า** (144s vs 62s) ⇒ retry น้อย = call น้อย = ตรงกับปัญหาที่กำลังแก้
สิ่งที่ทำให้ SwiftCap ได้ 0.91 คือ **รูปทรงการเรียก ไม่ใช่ตัวโมเดล**

### 4.2 ตาข่ายกันตก (ทำงานเฉพาะตอนพัง ไม่แตะ happy path)

| | สิ่งที่ทำ |
|---|---|
| H1 | retry มี **jitter** (16 call ไม่ตื่นจาก 429 พร้อมกันแล้วชนซ้ำ) + **รู้เวลาที่เหลือ** (ไม่ sleep เข้าไปใน retry ที่ยังไงก็ไม่ทัน) |
| H2 | **stagger** เวลาเริ่มของแต่ละคลิป — ไม่ใช่ semaphore |
| H3 | **ขั้นบันไดกู้** ก่อนถึง generic: emergency 1 เฟรม → re-voice ข้อเท็จจริงจากสไตล์ที่สำเร็จ |
| H4 | telemetry `[health] fallback_clips=N` — ทำให้ "คลิปตาย" มองเห็นได้ |
| H5 | **ดึงเฟรมตรงจาก URL** ด้วย HTTP range request เมื่อ download ตาย (เดิม download ตาย = เสียทั้งคลิป 4 สไตล์) |

ถอด `QWEN_DIRECT_MAX_INFLIGHT=3` ออกจาก Dockerfile ด้วย — เป็น semaphore ของ r6 ที่ตายไปแล้ว
แต่ยังนอนรออยู่ (ดูข้อ 6.4)

### 4.3 ผลวัด

| การทดสอบ | r1 | v8 / v7-hard |
|---|---|---|
| judge ที่แยกของออก (gemini-2.5-flash) | 0.9108 | **0.9075** ← เท่ากัน |

⇒ **ลด API call 4 เท่าโดยไม่เสียคุณภาพ caption เลย**

## 5. สิ่งที่ "ไม่ได้" ทำให้คะแนนขึ้น (สำคัญพอ ๆ กัน)

- **ไม่ได้แก้ prompt เลยสักตัว** — ลองเกลา `humorous_non_tech` (สไตล์ที่อ่อนสุดทุก judge) แล้ว
  **มันแย่ลง**: สไตล์เป้าหมายขยับแค่ 0.882 → 0.883 แต่รวมตก **0.9329 → 0.9246** ⇒ **revert ทิ้ง**
  ถ้าดันทุรังส่งไป อาจไม่ได้ 0.91
- **gemini-3.5-flash ที่จ่ายไป ~$1.2 ไม่ได้ช่วยตัดสินใจอะไรเลย** — มัน **saturate**:
  ให้ build ที่บอร์ดจริงให้ 0.66 ถึง **0.9875** และแจก 1.00 ~90% ของ caption
  (แก้ด้วย `EVAL_STRICT=true` = rubric แบบ anchor สเกล ดึงกลับมาที่ 0.9329 ใช้ได้)

## 6. บทเรียน

### 6.1 คะแนนตกฮวบ → ดู digest กับ timestamp ก่อน อย่าเพิ่งแตะโค้ด
สองคะแนน (0.77, 0.69) ถูกตรวจจาก **image ผิดตัว** เพราะ submission อ้าง tag `:latest`
ใครก็ตามที่ push ทับ = เปลี่ยนของที่ถูกตรวจทันที และ **ระบบ pull ตอน eval ไม่ใช่ตอน submit**
⇒ คะแนนที่เห็นวันนี้อาจเป็นของ image เมื่อ 3 ชั่วโมงก่อน

**กติกา:** ทุก build ทดลอง = tag ของตัวเอง · `:latest` ขยับเมื่อตกลงกันแล้วเท่านั้น ·
เทียบคะแนนกับ digest ที่ `:latest` ชี้อยู่ ณ เวลา (scored − eval latency)

### 6.2 ถามว่า "เลขนี้อธิบายด้วย *ของพังกี่ชิ้น* ได้ไหม"
คะแนนรวมที่ตกแบบเรียบ ๆ = คุณภาพแย่ลง
คะแนนที่ตกไปเกาะเลขคณิตของ "N หน่วยได้ศูนย์" = **มีของพัง** — คนละวิธีแก้กันคนละขั้ว
สูตร `baseline − (k/N) × Δ` ใช้เวลาคิด 30 วินาที และมันชี้ถูก

### 6.3 local judge วัด "คุณภาพ caption" ไม่ได้วัด "คลิปรอดไหม"
local run เรา `fallback_clips=0` เสมอ ⇒ judge ตรวจ 60 caption ที่สำเร็จหมด
แต่บอร์ดตรวจ **caption × คลิปที่รอด** ⇒ ต่อให้ปรับ judge ให้โหดจน r1 ได้ 0.70 ก็ยัง**หาบั๊กไม่เจอ**
สิ่งที่ควรทำคือ **fault injection** (จำลอง 429) ไม่ใช่ปรับ rubric

### 6.4 throttle "จังหวะเริ่ม" ได้ แต่ห้ามคาป "ปริมาณงานที่ไหลได้"
r6 ใส่ semaphore จำกัด in-flight call ไว้ **3** ทั้งที่ pipeline ต้องการ **16**
บนเครื่องตรวจที่ช้ากว่า มันเลยไล่ไม่ทัน ชน time cutoff → คลิปตก fallback → **0.69**
**stagger เวลาเริ่ม ≠ cap throughput** — อันแรกปลอดภัย อันหลังฆ่าเรามาแล้ว

### 6.5 จำนวน API call คือ "ตัวแปรความน่าเชื่อถือ" ไม่ใช่แค่ "ตัวแปรค่าใช้จ่าย"
ในสนามที่มี rate limit + เวลาจำกัด ทุก call คือโอกาสพังอีกหนึ่งครั้ง
ทีมที่ชนะไม่ได้ prompt ดีกว่า — **เขายิงน้อยกว่า**

### 6.6 fallback ที่ "ไม่มีวันว่าง" ซ่อนความพังไว้ — ต้องมี telemetry
generic fallback ทำให้ output ดูสมบูรณ์ (ครบ 4 สไตล์ ไม่มีช่องว่าง) แต่ accuracy = 0
ถ้าไม่นับมันออกมาเป็นตัวเลข เราจะไม่มีทางรู้ว่าเสียคลิปไป 4 ⇒ `[health] fallback_clips=N`

### 6.7 judge ที่ใจดีเกินไป แย่กว่าไม่มี judge
มันฟอก caption อ่อน ๆ ให้ดูสมบูรณ์แบบ ถ้า judge ให้ 1.00 กับเกือบทุกอัน = **มันไม่มีข้อมูล**
วิธีแก้: anchor สเกล (caption ธรรมดาเริ่มที่ 0.7, 1.0 ต้องหาเหตุผลมาให้ได้), บังคับให้ไล่
claim ที่ไม่มีในภาพ **ก่อน** ให้คะแนน

### 6.8 ใกล้เดดไลน์ = prompt คือของที่ห้ามแตะที่สุด
variance สูงสุด, EV ต่ำสุด, วัดผลไม่ทัน — และเรามีหลักฐานว่ามันทำให้แย่ลงจริง (ข้อ 5)
เปลี่ยน **สถาปัตยกรรมที่วัดได้** ดีกว่าเปลี่ยน **ถ้อยคำที่วัดไม่ได้**
